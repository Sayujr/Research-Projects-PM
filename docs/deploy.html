Deploy

Getting this live for your lab. GitHub Pages is free and enough for most; Cloudflare Access adds a password gate; custom domain is optional.
Total time: 30–60 min for a lab-ready deploy. You can ship the free public version in 5 min and layer auth + domain on later.

Choose a privacy level

1. Public repo + public Pages free · 5 min

Fastest. Good for demos and open-science labs. Anyone with the URL sees your projects + progress. Collaborator names appear in people.md.

2. Public repo + Cloudflare Access gate free · 20 min

Public URL, but Cloudflare Access challenges for email before showing anything. You maintain an allowlist. Recommended for most labs.

3. Private repo + Pages GitHub Pro / Team · 5 min

Repo itself is private (your lab only). Pages can still be public or private (Pro+). Simplest model if collaborator names must stay internal. Costs $4/user/month on Team.

4. Self-hosted on a lab server sysadmin time

Required if institutional policy bars cloud storage of unpublished research. Use Forgejo or Gitea + a static-site server (nginx/caddy). Same markdown source, same scripts. The GitHub Actions become Forgejo Actions or a cron job.

Path 1 — public Pages (you already did this)

1

Fork or clone this repo under your account

Either fork or clone and push to a new repo of your own.

2

Enable GitHub Pages

Repo Settings → Pages. Source: Deploy from a branch. Branch: main. Folder: /docs. Save.

First build takes ~30s. URL appears at the top of the Pages page — format https://YOURUSER.github.io/YOURREPO/.

3

Update the bootstrap prompt URL

Edit docs/setup.html — replace the repo URL in the prompt block with yours. Rebuild and push; Pages auto-updates.

Path 2 — add Cloudflare Access

One-time setup, free for up to 50 users.

1

Put the site behind a Cloudflare domain

Options:

  • Bring your own domain — e.g. lab-pm.yourlab.edu. Point a CNAME at YOURUSER.github.io. Verify in GitHub Pages settings.
  • Use a Cloudflare-managed subdomain — free with any Cloudflare account.
2

Create a Cloudflare Access application

Cloudflare dashboard → Zero Trust → Access → Applications → Add.

Type: Self-hosted. Domain: your Pages domain. Path: / (everything).

3

Add an access policy

Policy name: Lab team. Action: Allow. Include rules:

  • Emails: list each lab member's email
  • Email domain: yourlab.edu (covers everyone at the institution)

Authentication methods: One-time PIN via email is fine and requires no identity provider setup. Google SSO works if your lab uses Google Workspace.

4

Test

Open the URL in an incognito window. Cloudflare should challenge for email → send PIN → enter PIN → site loads. Logged-in users stay authenticated for the session duration you configured (default 24h).

If you also want the repo source to be private (not just the site), you'll need GitHub Pro or Team. The site can live on a public Pages while the repo is private — but be aware the generated data.json contains whatever's in the markdown.

Post-deploy: multi-user scope

If multiple lab members will access the same site, a per-user view helps them focus:

  • Append ?user=sara to any page URL — the dashboard and Plan Board filter to projects she owns or collaborates on.
  • Bookmark the URL with the parameter and they land on "their view" every time.

Filter is a lightweight UI convenience, not a permission boundary. If you need real role-based access, layer a CODEOWNERS file + GitHub branch protection on top.

Migrating from Notion / Asana / Todoist

There isn't a one-click importer. The manual path takes 1–2 hours for a typical lab:

  1. Export your existing tool to CSV / markdown (all three support this).
  2. Project per project, paste contents into projects/<id>.md under the relevant section headings. Fill in the frontmatter (goal_model, deadline, collaborators).
  3. Dump completed tasks into progress/YYYY-MM.md as one line each.
  4. Run /standup in Claude. It'll point out what's missing.

If you have a big migration and want a scripted importer for your specific source, open an issue with a sample export and we'll write one.

You now own the data. The repo is portable — clone, back up, move to another host any time.
